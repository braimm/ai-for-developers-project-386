На этом шаге мы реализуем серверную часть приложения по контракту, который зафиксировали ранее.
Реализация должна идти от контракта, а не от выбранного фреймворка.

Требования:
1. Бэкенд предоставляет API по спецификации контракта из шага проектирования.
2. API бэкенда предназначен для отдельного фронтенд-клиента.
3. Основные бизнес-правила бронирования реализуются на стороне бэкенда.
4. Для этого шага отдельная база данных не нужна. Достаточно простого хранилища в памяти: после перезапуска сервиса данные могут сбрасываться.

Задачи:
1. Реализуйте API по созданной спецификации контракта.
2. Сделайте обработку ситуации, когда выбранный слот уже занят.

Результат
1. Eсть рабочий бэкенд с хранением данных.
