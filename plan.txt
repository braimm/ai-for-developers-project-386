Стек проекта

Backend:
- FastAPI
- Pydantic для схем
- Uvicorn для запуска
- pytest для тестов API
- httpx или TestClient для интеграционных тестов
- ruff для линтинга

API contract:
- TypeSpec как источник истины
- генерация OpenAPI
- backend реализует контракт
- frontend опирается на контракт при интеграции

Frontend:
- Vite
- TypeScript
- React
- Mantine для UI
- React Router, если будут отдельные страницы owner/public
- fetch или axios
- Рекомендуемый вариант: обычный fetch, если API несложный

Хранение данных:
- Рекомендуемый вариант: SQLite + SQLAlchemy
- Причины:
  - есть реальные бронирования и правило конфликта по времени
  - проще сразу проектировать модели нормально
  - не придется потом переписывать слой хранения

Маршруты фронта:
- / или /book — публичный каталог типов событий
- /events/:id — выбор слота
- /bookings/success — подтверждение брони
- /owner/event-types — управление типами событий
- /owner/bookings — список предстоящих встреч

План работ:
1. Уточнить инфраструктурные решения по backend и frontend.
2. Дожать TypeSpec под выбранную модель данных и страницы.
3. Поднять backend-структуру FastAPI:
   - routers
   - schemas
   - models
   - services
   - repositories
4. Реализовать API владельца и гостя.
5. Добавить проверку правила занятости слота.
6. Поднять frontend на Vite + React + Mantine.
7. Собрать страницы гостя и владельца.
8. Подключить frontend к backend.
9. Добавить тесты на ключевые сценарии.

Нужно решить перед реализацией:
1. База сразу SQLite или сначала временно in-memory?
2. Нужен ли React Router, или делаем маленький SPA без маршрутизации?
3. Хотим ли сразу разделить проект на backend/ и frontend/, или оставить backend в корне, а frontend в frontend/?

Рекомендуемая структура:
- backend/
- frontend/
- typespec/
